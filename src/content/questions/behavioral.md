## Почему спрашивают про пет-проекты?

Пет-проекты показывают:
- **Мотивацию** — вы учитесь и развиваетесь вне работы
- **Практические навыки** — можете применить знания на практике
- **Инициативность** — не ждете, когда вам дадут задачу
- **Интересы** — что вам действительно нравится в разработке
- **Способность довести дело до конца** — завершенный проект важнее незаконченного

## Методика STAR

Используйте формат STAR для структурированных ответов:

- **S**ituation — Ситуация (контекст)
- **T**ask — Задача (что нужно было сделать)
- **A**ction — Действие (что вы сделали)
- **R**esult — Результат (чего достигли)

### Пример ответа по STAR

**Вопрос**: Расскажите о сложной технической проблеме, которую вы решили.

**Ответ**:
- **S**: В проекте e-commerce страница каталога загружалась 8 секунд при 1000+ товарах
- **T**: Нужно было ускорить загрузку до 2 секунд без потери функциональности
- **A**: Внедрил виртуальный скроллинг, оптимизировал запросы к API (добавил пагинацию), использовал мемоизацию для фильтров
- **R**: Время загрузки сократилось до 1.5 секунд, bounce rate снизился на 30%

## Вопросы о работе в команде

### Конфликты и сотрудничество

**Расскажите о конфликте с коллегой. Как вы его разрешили?**
- Опишите конкретную ситуацию
- Покажите эмпатию и понимание другой стороны
- Объясните, как нашли компромисс
- Подчеркните, что вынесли из этого опыта

**Как вы работаете с людьми, с которыми не согласны?**
- Слушаете их аргументы
- Приводите свои с примерами
- Ищете data-driven решения
- Готовы принять решение команды

**Опишите ситуацию, когда вам пришлось работать с трудным человеком.**
- Фокус на профессионализме
- Как адаптировали стиль коммуникации
- Что сделали для продуктивной работы

### Командная работа

**Расскажите о своей роли в команде.**
- Какие задачи обычно берете
- Как помогаете другим
- Примеры менторства или code review

**Как вы онбордите нового члена команды?**
- Документация и знакомство с кодбейзом
- Парное программирование
- Постепенное усложнение задач

**Опишите успешный проект, где вы работали в команде.**
- Ваша конкретная роль
- Как координировались
- Какой результат достигли

## Советы по подготовке

### До интервью

1. **Подготовьте 5-7 историй** из опыта работы
2. **Используйте формат STAR** для структуры
3. **Будьте конкретны** — цифры, метрики, результаты
4. **Исследуйте компанию** — продукт, культура, технологии
5. **Подготовьте вопросы** для интервьюера

### Во время интервью

1. **Слушайте вопрос внимательно** — можно попросить уточнить
2. **Думайте вслух** — покажите процесс мышления
3. **Будьте честны** — лучше признать незнание, чем врать
4. **Показывайте энтузиазм** — интерес к компании и роли
5. **Задавайте вопросы** — это диалог, а не допрос

### Что говорить

✅ **Хорошо:**
- "Я проанализировал проблему и..."
- "Мы с командой решили..."
- "Я изучил документацию и..."
- "Результат превысил ожидания на 20%"

❌ **Плохо:**
- "Это было легко"
- "Я всегда прав"
- "Мой коллега был некомпетентен"
- "Я не помню деталей"

### Красные флаги (чего избегать)

- Обвинение других в неудачах
- Отсутствие конкретики
- Неспособность признать ошибки
- Негатив о прошлых работодателях
- Отсутствие примеров

## Вопросы для интервьюера

Подготовьте вопросы, которые покажут ваш интерес:

### О команде
- Как организована команда разработки?
- Как проходит процесс code review?
- Какие инструменты используете для коммуникации?

### О процессах
- Как выглядит типичный спринт?
- Как принимаются технические решения?
- Как измеряете успех проектов?

### О росте
- Какие возможности для обучения?
- Есть ли бюджет на конференции/курсы?
- Как выглядит карьерный рост?

### О культуре
- Как бы вы описали культуру команды?
- Как поддерживаете work-life balance?
- Что вам больше всего нравится в работе здесь?

## Примеры хороших ответов

### Про пет-проект

"Я создал приложение для трекинга привычек на React и TypeScript. Хотел изучить React Query и state management с Zustand. Проект помог мне понять, как оптимизировать повторные рендеры и работать с кешированием данных. Приложение использую сам каждый день, и несколько друзей тоже начали им пользоваться. Код выложил на GitHub с подробным README."

### Про сложную задачу

"В нашем приложении была проблема с memory leak в списке сообщений. Страница начинала тормозить после 10 минут использования. Я использовал Chrome DevTools Memory Profiler, нашел, что event listeners не очищались при размонтировании компонентов. Добавил cleanup функции в useEffect, написал тест, который проверяет отсутствие утечек. После фикса приложение могло работать часами без проблем."

### Про конфликт

"У нас с тимлидом были разные мнения по архитектуре нового фичи. Я предлагал микрофронтенды, он — монолит. Я подготовил документ с pros/cons обоих подходов, включая метрики производительности и сложность поддержки. Мы обсудили это с командой, и решили начать с монолита, но с модульной архитектурой, чтобы при необходимости можно было разделить. Это оказалось правильным решением — мы быстрее запустились, а модульность помогла поддерживать код."
